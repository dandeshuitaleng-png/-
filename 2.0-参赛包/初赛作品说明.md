# 初赛作品说明：今天吃什么 2.0

## 作品基本信息

| 项目 | 内容 |
| --- | --- |
| 作品名称 | 今天吃什么 2.0 |
| 作品类型 | iOS 饮食决策助手 |
| 目标平台 | iOS，后续可扩展 iPadOS / visionOS |
| 目标用户 | 高校学生、办公园区年轻人、午晚餐选择困难用户 |
| 核心价值 | 结合位置、时间、预算、忌口、距离、出餐速度和历史偏好，在 10 秒内给出一个有理由、可执行的一餐选择 |

## 一、问题背景与用户分析

高校学生和年轻上班族每天都会面对“吃什么”的小决策。传统外卖和点评平台提供大量商家、折扣和评价，但默认用户已经知道想吃什么；当用户处于饥饿、赶课、午休时间短或多人意见不一致的场景时，过多选项反而会增加决策成本。今天吃什么 面向这类“需要快速决定，而不是继续浏览”的用户，将一餐选择拆成预算、距离、忌口、出餐速度和健康目标几个关键因素，用更少操作给出可执行推荐。

## 二、相关竞品分析

美团、饿了么和大众点评的优势是商家覆盖、交易链路和评价体系，适合“我想吃火锅，找一家店”的搜索场景；小红书适合发现新店和内容种草，但决策链路更长。今天吃什么 2.0 的差异化不是替代交易平台，而是放在决策前一步：当用户没有明确目标时，先用两题问答和偏好筛选缩小范围，再给出一个主推荐和少量备选。产品强调“少看、快选、少后悔”，避免把用户再次推回无限列表。

## 三、可行性分析

2.0 可以分阶段实现。初赛阶段采用 iOS 高保真原型和 SwiftUI MVP 骨架，验证完整用户路径：打开 App、使用系统定位或手动选择校园区域、一键推荐或回答两题、查看一个主推荐和两个备选、理解推荐理由、收藏或标记已吃、从历史记录再次推荐。数据层继承 1.0 的筛选、逐级放宽、避免重复、收藏权重和自定义餐饮数据能力，并使用约 30 条校园周边样本数据验证推荐链路。复赛阶段可接入 CoreLocation、MapKit 或第三方餐饮数据源，通过本地历史记录继续提升个性化权重。

## 四、App 创新点

今天吃什么 2.0 的创新点在于把“随机”升级为“有约束的轻决策”。第一，推荐不是纯随机，而是结合位置、时间、预算、距离、忌口、出餐速度、健康标签和历史行为，让结果更可信。第二，产品提供两种入口：时间紧时一键推荐，想表达当下需求时只回答两题，兼顾效率和参与感。第三，结果页不只展示店铺，还解释推荐理由，并严格控制为一个主推荐和两个备选，避免重新制造选择负担。第四，定位授权不是使用门槛；拒绝授权后可以手动选择校园区域。第五，收藏、标记已吃、近期不推荐和历史回用共同形成轻量个性化闭环。

## 五、应用前景

该产品适合从高校和办公园区切入，因为这些场景餐饮选择集中、决策频次高、用户时间敏感。短期可以做成个人工具，帮助用户减少午晚餐纠结；中期可以接入校园周边餐厅、食堂窗口、优惠信息和营养标签；长期可以扩展到多人投票、低碳饮食、预算管理和健康计划。与大平台不同，今天吃什么 2.0 的价值在于更靠近“决定吃什么”的瞬间，适合作为轻量、高频、可持续迭代的移动应用。

## 六、初赛演示路径

1. 首次启动完成手机号验证码或 Apple ID 原型注册，设置昵称、头像、性别、生日和兴趣偏好，再通过新手引导进入首页。
2. 首页默认使用“一键推荐”，直接结合当前区域、用餐时间和长期偏好生成结果。
3. 结果页给出一个主推荐、两个附近备选、预算、距离、出餐速度和明确的推荐理由。
4. 切换“两题模式”，回答“这顿更看重什么”和“想吃什么方向”，生成更贴合当下的结果。
5. 将满意结果收藏或标记为已吃；选择“近期不推荐”后，后续一轮会主动避开该餐饮点。
6. 在“我的”中从历史记录再次推荐，并演示重新体验引导、定位拒绝后的区域选择和餐饮数据维护。
