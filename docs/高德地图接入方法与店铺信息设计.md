# 高德地图接入方法与店铺信息设计

> 项目场景：校园餐饮推荐 / 长沙师范学院安沙校区附近  
> 产品目标：在「吃什么 App」中接入真实周边店铺、步行距离、路线导航，并结合今日偏好与长期偏好做翻牌推荐。  
> 核心原则：高德负责真实地理数据，你的系统负责餐饮理解、偏好匹配和翻牌推荐。

---

## 1. 接入目标

「吃什么 App」接入高德地图后，完整流程应该是：

```text
获取当前位置
↓
调用高德周边 POI 搜索
↓
拿到附近餐饮店铺
↓
清洗成 App 可用的餐饮数据
↓
结合今日偏好做推荐
↓
推荐卡片支持收藏、详情、导航
```

需要明确区分数据来源：

| 能力 | 来源 |
|---|---|
| 店名、地址、经纬度、距离、分类、营业时间、人均、评分 | 高德 POI |
| 是否辣、健康标签、出餐速度、推荐理由、收藏状态 | 自有业务数据 |
| 步行距离、预计时间、路线步骤 | 高德路径规划 |
| 翻牌推荐逻辑 | 自有推荐算法 |

## 2. 需要接入的高德能力

### 2.1 Web 服务 Key

先在高德开放平台创建应用并添加 Key，服务平台选择：

```text
Web 服务
```

后端保存 Key：

```env
AMAP_WEB_KEY=你的高德Web服务Key
```

### 2.2 注意事项

不要把高德 Web 服务 Key 直接写死在前端。

建议结构：

```text
前端 App
↓
你的后端接口
↓
高德 Web 服务 API
```

原因：

- 避免 Key 暴露
- 方便做缓存
- 方便做数据清洗
- 方便统一处理接口失败
- 方便把高德数据转成自己的店铺数据结构

## 3. 店铺搜索接口

### 3.1 周边餐饮搜索

用于查询用户附近真实餐饮店铺。

接口：

```http
GET https://restapi.amap.com/v5/place/around
```

核心参数：

| 参数 | 说明 | 建议值 |
|---|---|---|
| key | 高德 Web 服务 Key | 后端配置 |
| location | 中心点坐标，经度在前、纬度在后 | 用户当前位置 |
| radius | 搜索半径，单位米 | 1000 / 1500 / 2000 |
| types | POI 类型 | 050000，餐饮服务 |
| sortrule | 排序规则 | distance |
| show_fields | 扩展字段 | business,navi,photos |
| page_size | 每页数量 | 20 |
| page_num | 页码 | 1 |

请求示例：

```http
GET https://restapi.amap.com/v5/place/around?key=AMAP_KEY&location=113.xxxxxx,28.xxxxxx&radius=1500&types=050000&sortrule=distance&show_fields=business,navi,photos&page_size=20&page_num=1
```

### 3.2 坐标顺序

高德坐标顺序必须是：

```text
经度,纬度
```

不要写成：

```text
纬度,经度
```

否则位置会严重偏移。

## 4. 根据学校地址获取坐标

如果用户没有授权定位，可以先用学校地址做默认坐标。

接口：

```http
GET https://restapi.amap.com/v3/geocode/geo
```

请求示例：

```http
GET https://restapi.amap.com/v3/geocode/geo?key=AMAP_KEY&address=长沙市长沙县安沙镇万花园路9号&city=长沙
```

建议逻辑：

```text
有用户定位
↓
用用户定位

没有用户定位
↓
用长沙师范学院安沙校区默认坐标

定位失败
↓
提示用户手动选择校区
```

## 5. 店铺详情接口

周边搜索拿到 POI 后，可以用 POI ID 查询详情。

接口：

```http
GET https://restapi.amap.com/v5/place/detail
```

请求示例：

```http
GET https://restapi.amap.com/v5/place/detail?key=AMAP_KEY&id=POI_ID&show_fields=business,navi,photos
```

使用场景：

- 推荐卡片点击详情
- 首次缓存店铺数据
- 刷新店铺营业时间、人均、评分、图片
- 补充导航入口坐标

如果传入：

| 高德字段 | 含义 | App 用法 |

| opentime_today | 今日营业时间 | 营业时段 |

### 7.1 字段兜底规则

高德字段不一定每家店都有，所以需要兜底。

```text
人均没有
↓
用自维护估算价格

营业时间没有
↓
显示“营业时间待确认”

图片没有
↓
用默认餐饮图

评分没有
↓
不展示评分

入口坐标没有
↓
用 POI location
```

## 8. 导航与路线接口

### 8.1 步行路线规划

接口：

```http
GET https://restapi.amap.com/v3/direction/walking
```

请求参数：

| 参数 | 说明 |
|---|---|
| key | 高德 Web 服务 Key |
| origin | 起点，经度,纬度 |
| destination | 终点，经度,纬度 |
| origin_id | 起点 POI ID，可选 |
| destination_id | 终点 POI ID，可选 |
| output | JSON |

请求示例：

```http
GET https://restapi.amap.com/v3/direction/walking?key=AMAP_KEY&origin=113.xxxxxx,28.xxxxxx&destination=113.xxxxxx,28.xxxxxx
```

### 8.2 路线返回字段

| 高德字段 | App 用法 |
|---|---|
| route.paths[0].distance | 步行距离 |
| route.paths[0].duration | 步行时间，秒 |
| route.paths[0].steps | 路线步骤 |
| steps[].instruction | 每段路线说明 |
| steps[].polyline | 地图路线折线 |

App 展示示例：

```text
距离：0.7km
步行：约9分钟
路线：校门口 → 万花园路 → 店铺
```

## 11. 店铺信息清洗规则

高德返回的是地图 POI，不是可直接用于推荐的餐饮数据，必须清洗。

### 11.1 过滤规则

```text
只保留 types=050000 餐饮服务
排除过远店铺
排除名称明显不是餐饮的 POI
排除重复店铺
排除缺少坐标的 POI
```

### 11.2 去重规则

优先按：

```text
sourcePoiId
```

其次按：

```text
店名 + 地址 + 经纬度距离小于50米
```

### 11.3 品类归类规则

把高德的 `type` 转成 App 内部品类。

| 高德 type 可能值 | App 品类 |
|---|---|
| 餐饮服务;中餐厅 | 湘菜 / 家常菜 |
| 餐饮服务;快餐厅 | 快餐 / 盖饭 |
| 餐饮服务;小吃快餐店 | 小吃 / 粉面 |
| 餐饮服务;咖啡厅 | 咖啡 / 轻食 |
| 餐饮服务;冷饮店 | 茶饮 / 果饮 |
| 餐饮服务;烧烤店 | 烧烤 / 夜宵 |
| 餐饮服务;外国餐厅 | 韩式 / 西式 / 其他 |

## 15. 推荐算法基础版

MVP 阶段不要做复杂算法，用打分规则即可。

```text
距离匹配：30分
价格匹配：20分
品类匹配：20分
口味匹配：15分
出餐速度：10分
健康标签：5分
```

示例代码：

```ts
function scorePlace(place, todayPreferences, userProfile) {
  let score = 0

  if (place.distanceMeters <= userProfile.defaultDistanceMeters) score += 30
  if (place.avgPrice <= userProfile.defaultBudget) score += 20
  if (place.category.includes(todayPreferences.foodType)) score += 20
  if (tasteMatched(place.spicyLevel, todayPreferences.taste)) score += 15
  if (todayPreferences.status === '赶时间' && place.speedLevel === '快') score += 10
  if (todayPreferences.taste === '清淡点' && place.healthTags.includes('少油')) score += 5

  return score
}
```

## 16. MVP 接入顺序

不要一次性全做。建议按这个顺序：

```text
第 1 步：申请高德 Web 服务 Key
第 2 步：后端封装 /api/food/nearby
第 3 步：调用 place/around 获取周边餐饮
第 4 步：存入 food_places 表
第 5 步：补充自维护字段：辣度、人均、健康标签、出餐速度
第 6 步：决定页翻牌推荐使用真实店铺池
第 7 步：推荐卡片增加“去这里”
第 8 步：调用 direction/walking 获取步行时间和路线
第 9 步：收藏页读取 user_favorites
第 10 步：我的页维护长期偏好
```

## 17. 最终结论

高德地图接入后，系统分工应该是：

```text
高德负责：
真实店铺、地址、经纬度、距离、路线、步行时间

你的系统负责：
餐饮标签、辣度判断、健康标签、出餐速度、推荐理由、收藏、翻牌推荐
```

一句话总结：

> 先让 App 能找到真实附近店铺和真实路线，再用自己的标签系统把它变成“今天适合吃什么”的推荐。
