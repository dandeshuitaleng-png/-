# 「吃什么」App 1.0 产品分析报告

> 分析方法来自 PM Skills 库：`prd-development`、`user-story`、`customer-journey-map`、`problem-statement`、`prioritization-advisor`、`tam-sam-som-calculator`

---

## 一、产品定位

| 维度       | 详情                          |
| -------- | --------------------------- |
| **产品名称** | 今天吃什么                       |
| **平台**   | HarmonyOS (ArkTS)           |
| **版本**   | 1.0 (M6 结档)                 |
| **定位**   | 解决"今天吃什么"选择困难的快速决策工具        |
| **差异化**  | 随机推荐 + 智能筛选 + 翻牌互动，而非传统列表浏览 |
| **当前状态** | 模拟数据验证，真实定位/导航/签名后置 2.0     |

---

## 二、问题陈述 (Problem Statement)

**谁有这个问题？**
工作日午休时间有限的上班族、学生、对周边餐饮不熟悉的新居民。

**什么问题？**
在美团/大众点评等平台浏览时，面对大量选项陷入「选择困难」，反复纠结浪费时间。现有平台解决的是「**找到更多选项**」，而非「**快速做出决定**」。

**为什么痛？**
- 每次纠结 5-15 分钟，一个月的累计时间损失可观
- 同事群反复问「吃什么」成为社交摩擦
- 决策疲劳降低午餐满意度

**证据：**
- 「今天吃什么」是中文互联网高频搜索/讨论话题
- 美团/大众点评未覆盖该场景——它们假设用户知道想吃什么，只需找到对应店铺

## 三、核心用户故事

| 优先级 | 用户故事 |
|--------|----------|
| **P0** | 作为午休时间有限的白领，我要一键随机获取附近餐厅推荐，以便 10 秒内做出决定 |
| **P0** | 作为口味挑剔的用户，我要设置辣度/预算/距离筛选，以便结果不踩雷 |
| **P1** | 作为喜欢参与感的用户，我要通过回答 2 个问题缩小范围，以便结果更贴合当下心情 |
| **P1** | 作为老用户，我要收藏常去的品类和店铺，以便下次快速找到 |
| **P2** | 作为小吃摊爱好者，我要自己添加路边摊信息，以便不被系统限制 |

## 四、系统架构

```
┌─────────────────────────────────┐
│           Index.ets             │  ← 单页面多状态 (home/flip/result/favorites/form)
├─────────────────────────────────┤
│     RecommendationService       │  ← 核心推荐引擎：餐段判断、品类匹配、距离预算过滤
├─────────────────────────────────┤
│     RestaurantRepository        │  ← 数据仓库：聚合 Mock + 自定义数据
├─────────────────────────────────┤
│  CombinedRestaurantAdapter      │  ← 适配器模式：MockAdapter + CustomRestaurants
├─────────────────────────────────┤
│  AppContentService              │  ← 持久化层：收藏/自定义数据 CRUD
├─────────────────────────────────┤
│  PreferencesLocalStorageAdapter  │  ← 基于 HarmonyOS Preferences 存储
└─────────────────────────────────┘
```

### 架构优点
- 适配器模式 (`CombinedRestaurantAdapter`) 为将来接入真实 API 留了扩展点
- 推荐引擎采用 **级联放松策略**（距离→预算→辣度→问卷fallback），确保不会「查无结果」
- 依赖注入构造（`currentTimeProvider`, `randomNumberProvider`）使得测试可行

### 架构风险
- 所有 UI 逻辑集中于 `Index.ets`（1900+ 行），页面状态 7 个（home/flip/result/favorites/restaurantForm/categoryManager/categoryForm），后续维护成本高
- 没有路由抽象，页面切换全靠 `currentPage: string` 和 if/else 分支
- 缺少用户偏好 Profile 模型——目前筛选条件每次重置

## 五、推荐引擎深度分析

```
用户输入 → 确定餐段(早/午/晚) → 筛选品类(餐段+辣度+问卷)
         → 过滤店铺(距离+预算+营业状态)
         → 随机抽取品类 → 翻牌动画 → 展示结果(品类+Top3店铺)

如果 0 结果 → 逐级放松：
  距离限制 → 预算限制 → 辣度限制 → 问卷fallback
```

### 亮点
- 翻牌动画交互设计增加趣味性和「命运感」
- 「品类已曝光完自动重置」机制（cycleReset）避免重复
- 距离不足时自动扩大范围并提示用户

### 待优化
- 目前 **天气因素未实际接入**——开发日志中提到但代码中未实现
- 随机算法为纯均匀分布，无个性化权重（用户偏好、历史选择未纳入）
- 评分 (`rating`) 在 Mock 数据中存在但推荐中未使用

## 六、用户旅程地图 (Customer Journey Map)

| 阶段       | 触达     | 定位      | 选择模式           | 推荐     | 结果        | 回味         |
| -------- | ------ | ------- | -------------- | ------ | --------- | ---------- |
| **用户行为** | 打开 App | 等待定位/授权 | 选「直接抽取」或「回答两题」 | 看翻牌动画  | 查看品类+3家店铺 | 收藏/导航/再来一次 |
| **触点**   | 桌面图标   | 定位中加载页  | 模式切换+筛选面板      | 翻牌动画页  | 结果页+店铺卡片  | 收藏/导航按钮    |
| **情绪**   | 😐 好奇  | 😤 授权摩擦 | 🙂 选择          | 😄 期待  | 😊 满意     | 👍 留存      |
| **痛点**   | —      | 拒绝授权→退出 | 想搜特定菜系无入口      | 加载慢时焦虑 | 仅3家不够选    | 下次打开筛选重置   |

**最大摩擦点：定位授权。** 当前逻辑：拒绝→直接退出 App。这是一个高风险设计——大量用户习惯性拒绝定位权限。

## 七、竞品简析

| 维度   | 吃什么 App | 美团      | 大众点评    | 小红书      |
| ---- | ------- | ------- | ------- | -------- |
| 核心场景 | 快速决策    | 浏览下单    | 评价参考    | 种草推荐     |
| 交互模式 | 随机推荐    | 搜索+列表   | 搜索+评分   | 信息流      |
| 决策时长 | <10秒    | >60秒    | >120秒   | 无明确决策    |
| 用户心态 | 「随便吃啥」  | 「我要吃火锅」 | 「这家好不好」 | 「有什么好吃的」 |
| 覆盖范围 | 周边3km   | 全城      | 全城      | 全网       |

**竞争定位：美团做「搜」，吃什么做「选」。差异化明确，但市场容量受限。**

## 八、1.0 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| 定位权限拒绝率 | **高** | 拒绝→直接退出，将损失大量用户；建议增加「手动输入位置」降级方案 |
| 模拟数据局限 | **高** | 当前全部 Mock 数据，无真实餐馆，无法验证核心价值 |
| 单页架构 | **中** | 1900 行全部在一个文件，2.0 需拆分组件 |
| 无用户账号 | **中** | 换设备丢失所有收藏和自定义数据 |
| 无商业化路径 | **低** | 1.0 阶段正常，但需在 2.0 规划中考虑 |
| HarmonyOS 独享 | **中** | 用户基数远小于 Android/iOS |

## 九、2.0 建议优先级

按 Impact/Effort 矩阵：

| 优先级 | 功能 | Impact | Effort |
|--------|------|--------|--------|
| **P0** | 手动位置输入（权限拒绝降级） | 高 | 低 |

| **P1** | 组件拆分 + 路由重构 | 中 | 中 |
| **P1** | 个性化权重（历史选择、收藏偏好） | 高 | 中 |
| **P2** | 天气因素接入推荐算法 | 中 | 低 |
| **P2** | 搜索功能 | 中 | 中 |
| **P3** | 社交分享（「今天吃了这个」） | 低 | 低 |
| **P3** | iOS/Android 跨平台 | 高 | 高 |

## 总结

「吃什么」在红海市场中切了一个**差异化精准的微型场景**——不做搜索，做决策。

- **1.0 做对了什么**：推荐引擎架构扎实（级联放松 + 适配器扩展），翻牌动画有记忆点，品类-店铺二层推荐模型清晰
- **1.0 的短板**：Mock 数据无法验证真实 PMF，定位授权策略过于刚硬，单页架构不可持续
- **2.0 的关键战役**：真实数据接入 + 权限降级方案。这两个问题不解决，产品无法走出验证阶段

如果 2.0 能解决核心挑战，这是一个有明确用户价值的工具型产品。
