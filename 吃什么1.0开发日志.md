# 智能餐饮随机推荐 App 方案

## 1. 项目背景
用户在日常生活中经常面临“今天吃什么”的选择困难。现有的美团、大众点评等平台虽然提供餐馆信息和评分，但没有针对用户纠结场景的**快速决策工具**。本 App 旨在通过**随机推荐 + 个性化智能筛选 + 互动选择**，帮助用户快速决定今日餐饮方案。

## 2. 核心目标
- 快速解决用户“吃什么”的选择困扰。
- 提供有趣、互动、游戏化的选择体验。
- 结合用户偏好、附近餐馆、评分、距离及当天天气温度等因素，实现智能推荐。

## 3. 核心功能

### 3.1 一键随机推荐（卡片翻牌效果）
- 用户点击首页“今天吃什么”按钮。
- 系统显示一组餐饮卡片，通过翻牌动画揭晓最终推荐。
- 每张卡片展示：
  - 餐馆名称
  - 推荐菜品类型
  - 距离
  - 评分
  - 人均价格
  - 推荐理由
  - 一键导航功能

### 3.2 两种随机模式

#### 模式一：智能随机
- 适合用户不想动脑，快速获取推荐。
- 系统根据以下因素筛选餐馆后随机推荐：
  - 用户口味偏好
  - 附近餐馆和路边摊
  - 距离远近
  - 餐馆评分
  - 当前天气和温度
  - 人均价格
  - 营业状态
- 示例逻辑：
  - 高温天气 → 增加凉面、沙拉、饮品权重
  - 低温天气 → 增加火锅、砂锅、汤粉权重
  - 用户偏好辣味 → 提高川湘菜、麻辣烫权重

#### 模式二：选择随机
- 提供轻量化互动，让用户参与决策。
- 系统先给出几个小问题/选项，让用户缩小选择范围：
  1. 想吃热乎的还是清爽的？
  2. 预算范围是多少？
  3. 能走多远？
  4. 更想吃正餐、小吃、粉面还是甜品？
- 用户选择后，系统在匹配范围内随机生成推荐。
- 优势：增强趣味性和参与感，结果更贴合用户心情。

### 3.3 用户偏好管理
- 用户可设置口味、预算、忌口、常吃餐馆、收藏小摊。
- 偏好数据用于智能随机算法，提高推荐准确性。

### 3.4 附近餐馆与路边摊
- 基于地图 API（华为 Site Kit / 高德地图）获取餐馆信息，包括评分、营业时间、距离等。
- 用户可手动添加未被地图收录的路边摊、小吃点。

### 3.5 推荐算法示例
```
综合推荐分 = 距离*25% + 评分*25% + 用户口味偏好*25% + 天气适配*15% + 人均价格*10%
```
- 从评分前 5–10 个餐馆中随机选一个，保证既随机又合理。

## 4. 页面结构设计

### 4.1 首页
- 大按钮：一键随机“今天吃什么”
- 快速筛选选项：
  - 附近距离（500m/1km/不限）
  - 预算范围
  - 辣/不辣

### 4.2 推荐结果页
- 卡片翻牌动画显示餐馆和菜品信息
- 可操作：重新随机 / 收藏 / 导航 / 不喜欢

### 4.3 偏好页
- 我的口味、预算、忌口
- 常吃餐馆、收藏小摊、黑名单管理

### 4.4 地图页
- 附近餐馆和小吃点展示
- 用户可添加新地点
- 一键导航至目标餐馆

## 5. 差异化与亮点
- **卡片翻牌效果**：增强游戏化体验。
- **两种随机模式**：智能随机满足快速决策需求；选择随机增加趣味互动。
- **天气温度适配**：让推荐更贴合实际环境。
- **用户偏好管理**：保证个性化，提升满意度。
- **鸿蒙桌面卡片**：快速在桌面查看推荐，无需打开 App。

## 6. MVP 最小可行版本
1. 定位获取当前位置
2. 搜索附近餐馆（API 接口）
3. 用户口味和预算设置
4. 根据评分、距离、偏好筛选餐馆
5. 一键随机推荐（卡片翻牌）
6. 收藏 / 不喜欢 / 重新随机

后续迭代可增加：天气温度推荐、路边摊手动添加、转盘动画、多人投票、鸿蒙桌面卡片。

## 4. 地图适配层交接

### 4.1 架构总览

```text
Index.ets
  │
  └─ RecommendationService.setLocation(deviceLocation)
       │
       └─ RestaurantRepository.searchNearby({ location, radiusMeters })
            │
            └─ CombinedRestaurantAdapter
                 │
                 ├─ baseAdapter: RestaurantDataAdapter
                 │    ├─ 当前：MockRestaurantAdapter (24 家模拟店铺)
                 │    └─ 替换：RealMapAdapter (真实地图服务)
                 │
                 └─ customRestaurantProvider: () => Restaurant[]
                      └─ 用户自定义店铺，自动合并
```

### 4.2 接入真实地图服务的接口

```typescript
// RestaurantDataAdapter.ets — 真实地图服务必须实现此接口
export interface RestaurantDataAdapter {
  searchNearby(request: RestaurantSearchRequest): Promise<RestaurantSearchResult>;
}

// RestaurantSearchRequest
interface RestaurantSearchRequest {
  location: GeoPoint;    // 设备当前位置
  radiusMeters: number;  // 搜索半径 (默认 3000)
}

// RestaurantSearchResult
interface RestaurantSearchResult {
  status: RestaurantSearchStatus;  // SUCCESS 或 FAILURE
  restaurants: NearbyRestaurant[]; // 按距离排序
  errorMessage: string;
}

// NearbyRestaurant
interface NearbyRestaurant {
  restaurant: Restaurant;  // 需填充完整字段
  distanceMeters: number;  // 距当前位置的米数
}
```

### 4.3 接入步骤

1. 新建类实现 `RestaurantDataAdapter` 接口，例如 `SiteKitRestaurantAdapter`
2. 在 `searchNearby()` 中调用华为 Site Kit / 高德地图 API 搜索附近 POI
3. 将 API 返回的 POI 转换为 `Restaurant` 模型，需填充 `categoryIds` 字段
4. 计算每个 POI 到 `request.location` 的 Haversine 距离
5. 返回 `{ status: SUCCESS, restaurants: [...], errorMessage: '' }`
6. 在 `Index.ets` 中将 `CombinedRestaurantAdapter` 构造参数替换：

```typescript
// 当前 (Mock)
new CombinedRestaurantAdapter(new MockRestaurantAdapter(), () => this.customRestaurants)

// 替换为 (Site Kit)
new CombinedRestaurantAdapter(new SiteKitRestaurantAdapter(), () => this.customRestaurants)
```

### 4.4 Restaurant 模型字段映射

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 唯一标识，建议格式 `map-{poiId}` |
| `sourceType` | `RestaurantSourceType.MAP` | 是 | 标记为 `map` 来源 |
| `name` | string | 是 | 店铺名称 |
| `latitude` | number | 是 | 纬度 |
| `longitude` | number | 是 | 经度 |
| `categoryIds` | string[] | 是 | 关联品类 ID 列表，可通过地图分类映射 |
| `averagePrice` | number | 是 | 人均价格，若无数据填 0 |
| `rating` | number? | 否 | 评分 |
| `businessHours` | string? | 否 | 营业时间 |
| `isOpen` | boolean | 是 | 是否营业中 |

### 4.5 品类映射建议

真实地图 API 通常不返回菜品品类。建议在适配器中建立映射表：

```typescript
const CATEGORY_MAP: Record<string, string[]> = {
  'hotpot': ['hotpot'],
  'barbecue': ['barbecue'],
  'chinese': ['home-cooking'],
  'noodle': ['noodles'],
  'fast_food': ['quick-meal'],
  // ...
};
```

或使用 App 内置品类标签，让用户首次使用时手动为地图店铺补充品类。

## 3. 阻塞进入 M4 的问题

### 3.1 固定答案组合可能直接进入无结果

午餐和晚餐选择：

```text
清爽一点
→ 重口
```

当前内置品类中没有同时满足 `REFRESHING` 与 `SPICY` 的品类。推荐服务会保留问答答案作为硬条件，只放宽距离、预算和用户筛选辣度，因此该组合会直接返回无结果。

这不符合 M3 的核心目标：回答两题应该帮助用户降低纠结，而不是因为固定题目组合进入死路。

```text

→ 页面展示简短提示：按你的偏好选择不多，给你找了更接近的口味
```

该规则需要用户确认后，由研发 Agent 实现。

当前翻牌总时长符合要求，但每次切换固定等待 `180ms`。UI 规格要求前段快速切换、后段逐渐放慢，让最终揭晓更明确。

M3 新增的问答按钮高度、滑动阈值、翻牌卡片高度和动画时长仍散落在页面中。部分实现值也与 UI 规格不一致，例如问答按钮仍复用 `44vp` 高度，没有达到 UI 规格建议的至少 `48vp`。

## 3. 阻塞进入 M5 的问题

### 3.1 收藏店铺列表错误包含全部自定义店铺

当前收藏店铺列表的筛选条件为：

```text
已收藏店铺
或
自定义店铺
```

因此，用户只要创建了自定义店铺，即使没有收藏，也会在“收藏店铺”列表中出现。这会让“我的收藏”页面含义不准确，也会导致取消收藏后店铺仍留在列表中。

正确规则：

```text
收藏店铺列表只展示已收藏店铺
自定义店铺管理入口放在编辑和添加流程中
自定义店铺是否收藏，不影响它参与推荐
```

### 3.2 删除店铺缺少确认

编辑自定义店铺时，点击“删除店铺”会立即删除。M4 规格要求真正删除前提供确认，避免误操作。

### 3.3 删除未被使用的品类缺少确认

当前只有“仍被店铺使用”的品类会出现提示面板。未被使用的自定义品类点击删除后会立即执行。M4 规格要求两种情况都需要确认，只是提示文案不同。

## 4. 进入 M5 前需要收口的 UI 问题

# 今天吃什么 App - MVP 里程碑与 Agent 分工

## 1. 文档目标

本文件从产品经理视角拆分“今天吃什么”鸿蒙 App 的 MVP 开发过程，明确每个阶段需要验证的问题、各 Agent 的职责和进入下一阶段的条件。

## 2. 产品原则

- 首版目标不是堆叠功能，而是验证用户能否轻松、快速、有趣地决定“今天吃什么”。
- 每个里程碑都必须形成可体验、可验收的结果，不以“代码已经写完”作为完成标准。
- 先打通核心链路，再增加收藏、自定义和系统能力。
- 模拟地图数据与真实地图数据必须经过同一个适配层，避免后续接入真实地图时重写推荐逻辑。
- 1.0 使用模拟位置完成轻量核心体验；真实定位、权限申请和设备回归后置到 2.0。
- 首版不加入登录、跨设备同步、黑名单、天气推荐、桌面卡片、多人投票、历史记录和复杂反馈原因。

## 7. M2：直接抽取核心闭环

### 7.1 产品目标

### 7.2 产品 Agent 工作

- 实现距离、预算和辣度折叠筛选区域，默认收起。
- 实现无结果时按“距离 → 预算 → 辣度”逐步放宽条件。
- 实现结果页主文案，例如“今天吃火锅”。
- 展示距离最近的最多 3 家匹配店铺。
- 实现“再随机一次”，排除本轮已经出现过的品类。
- 候选耗尽后提示“换一轮口味”，并自动重置候选池。

### 7.4 UI Agent 工作

- 设计首页直接抽取状态。
- 设计折叠筛选区域的收起态、展开态和选中态。
- 设计结果页品类文案、店铺卡片和“再随机一次”按钮。
- 设计“附近选择不多，已经为你扩大范围”和“换一轮口味”提示。

### 7.5 验收门槛

- 用户无需配置即可完成一次直接抽取。
- 结果页正确显示一个品类和最多 3 家匹配店铺。
- 店铺卡片展示店名、距离、人均价格、评分、营业状态和数据标签。
- 再次抽取不会重复本轮已经揭晓过的品类。
- 条件过严时可以自动放宽并给出简短提示。

## 8. M3：回答两题与游戏化体验

### 8.1 产品目标

在不增加明显负担的前提下，让朋友聚餐时的选择过程更有参与感。

### 8.2 产品 Agent 工作

- 验收两题模式是否比直接抽取多出合理的参与感。
- 检查问题数量是否严格保持为 2 个。
- 检查翻牌动画是否服务于揭晓结果，而不是拖慢操作。

### 8.3 研发 Agent 工作

- 实现首页标签点击与左右滑动切换模式。
- 实现午餐、晚餐的固定两题。
- 实现早餐专用固定两题。
- 根据问题答案进一步过滤候选品类。
- 实现多张品类卡片快速切换动画。
- 控制动画总时长为 1-2 秒。

### 8.4 UI Agent 工作

- 设计“直接抽取 / 回答两题”模式标签与左右滑动反馈。
- 设计问答卡片、选项按钮、题数提示和返回修改方式。
- 设计多张品类卡片前快后慢的切换节奏。
- 检查翻牌结束后结果页的视觉重点是否明确。

### 8.5 验收门槛

- 用户可以通过点击标签或左右滑动切换模式。
- 回答两题后能够正常进入翻牌与结果页。
- 不同餐段展示正确问题。
- 问题答案能够影响候选品类。
- 翻牌动画时长符合 1-2 秒要求。

## 9. M4：收藏与自定义内容

### 9.1 产品目标

让用户保存有价值的结果，并补充地图数据难以覆盖的路边摊或非中式店铺。

### 9.2 产品 Agent 工作

- 验收收藏功能是否简单直接。
- 检查自定义内容是否复用统一模型。
- 控制表单字段数量，避免把轻量 App 做成复杂商户管理工具。

### 9.3 研发 Agent 工作

- 实现品类和店铺的收藏、取消收藏与本地持久化。
- 实现收藏页“收藏品类 / 收藏店铺”双标签。
- 在收藏店铺卡片中保留导航入口。
- 实现自定义店铺的添加、编辑和删除。
- 实现自定义品类的添加、编辑和删除。
- 添加店铺时支持选择多个品类，并允许当场创建新品类。
- 删除被使用中的品类前进行提示，仅解除关联，不删除店铺。
- 自定义店铺参与抽取、结果展示和距离排序。

### 9.4 UI Agent 工作

- 设计收藏页、空状态和取消收藏反馈。
- 设计“添加店铺 / 管理自定义品类”菜单。
- 设计轻量化表单、删除确认弹窗和关联品类提示。
- 为自定义店铺设计“自定义”标签。

### 9.5 验收门槛

- 重启 App 后收藏与自定义内容仍然存在。
- 自定义店铺和品类可以新增、编辑和删除。
- 自定义店铺能够参与抽取，并显示“自定义”标签。
- 删除品类不会误删关联店铺。

## 10. M5：定位、导航与系统异常

### 10.1 产品目标

补齐真实使用必须具备的系统能力，使 App 可以在真机上完成附近推荐和出发导航。

### 10.2 产品 Agent 工作

- 验收首次启动授权流程。
- 检查拒绝定位权限后的行为是否符合已确认规则。
- 检查导航是否足够直接，不引入 App 内地图页。

### 10.3 研发 Agent 工作

- 用户再次打开 App 时，若系统不再自动弹窗，引导前往系统设置开启权限。

### 10.4 UI Agent 工作

### 10.5 验收门槛

# 今天吃什么 App - MVP 产品 Spec

## 1. 文档目标

本文件用于指导“今天吃什么”轻量化鸿蒙 App 的首版开发与验收。

首版优先验证一个核心问题：学生或上班族与朋友聚餐时，能否通过轻量、有趣的随机过程快速决定“今天吃什么”。

## 2. 产品定位

### 2.1 核心用户

- 自己日常使用的学生或上班族。
- 优先覆盖与朋友聚餐时，由一个人操作手机完成选择的场景。

### 2.2 核心体验

- 轻量：避免复杂配置和冗余页面。
- 有趣：通过左右滑动切换模式、回答两题和快速翻牌增强参与感。
- 可行动：揭晓餐饮品类后，直接提供附近最多 3 家可导航店铺。

### 2.3 核心结果

结果页使用生活化文案：

```text
今天吃火锅
```

下方展示附近最多 3 家匹配店铺，而不是只给出抽象品类或强制指定唯一店铺。

## 3. MVP 功能范围

### 3.1 首版保留

- 根据当前时间自动判断早餐、午餐或晚餐。
- 首页左右滑动切换两种抽取模式。
- 可折叠的距离、预算和辣度筛选项。
- 多张品类卡片快速切换，最终揭晓一个结果。
- 结果页展示附近最多 3 家匹配店铺。
- 保留系统地图导航入口，作为试验性能力，不作为 1.0 正式交付门槛。
- 收藏品类和收藏店铺。
- 查看和取消收藏。
- 用户自定义添加、编辑和删除店铺或路边摊。
- 用户自定义添加、编辑和删除餐饮品类。
- 本地保存收藏与自定义内容。
- 使用模拟地图数据，并为后续接入真实地图服务保留统一适配层。

### 3.2 首版不做

- 真实定位、定位权限申请、拒绝授权强制退出和系统设置引导，统一后置到 2.0。
- 登录与账号体系。
- 跨设备同步。
- 黑名单。
- 天气推荐。
- 鸿蒙桌面卡片。
- 多人投票。
- “太贵”“太远”等细分反馈原因。
- 历史抽取记录页面。
- App 内店铺详情页。
- App 内地图页面。
- 自定义店铺地图选点。
- 收藏品类对抽取概率的加权。

## 4. 用餐时段

系统在后台根据设备当前时间判断用餐时段，不在首页额外展示提示。

| 时间范围 | 用餐时段 |
| --- | --- |
| 05:00-10:00 | 早餐 |
| 10:00-16:00 | 午餐 |
| 16:00-次日 05:00 | 晚餐 |

晚间夜宵场景统一归入晚餐，不增加额外分类。

## 5. 首页与两种模式

### 5.1 页面结构

首页为单页结构，不使用底部导航栏。

- 顶部右侧：收藏入口。
- 中部：当前抽取模式的主操作区域。
- 模式切换：支持左右滑动，同时保留可点击标签，避免用户无法发现另一个模式。
- 底部或主按钮附近：折叠筛选入口。

两个模式标签：

```text
直接抽取
回答两题
```

### 5.2 模式一：直接抽取

适用于希望快速获得结果的用户。

流程：

```text
获取附近营业中的店铺
→ 汇总店铺关联品类
→ 按当前用餐时段过滤
→ 应用用户主动展开并选择的筛选条件
→ 排除本轮已经揭晓过的品类
→ 随机抽取一个品类
→ 播放翻牌动画
→ 展示结果与附近店铺
```

### 5.3 模式二：回答两题

适用于希望参与决策过程的用户。

流程：

```text
获取附近营业中的店铺
→ 汇总店铺关联品类
→ 按当前用餐时段过滤
→ 提问两个问题
→ 应用问题答案和用户主动选择的筛选条件
→ 排除本轮已经揭晓过的品类
→ 随机抽取一个品类
→ 播放翻牌动画
→ 展示结果与附近店铺
```

午餐和晚餐固定问题：

```text
想吃热乎的还是清爽的？
今天想吃清淡、微辣还是重口？
```

早餐固定问题：

```text
想吃热食还是方便带走的？
想吃清淡的还是更管饱的？
```

首版使用固定问题，后续再考虑动态题库。

### 5.4 折叠筛选项

筛选条件默认收起，不打扰用户快速抽取。用户主动展开后可选择：

| 筛选项 | 选项 |
| --- | --- |
| 距离 | 500 米内 / 1 公里内 / 3 公里内 / 不限 |
| 人均预算 | 30 元内 / 30-60 元 / 60-100 元 / 不限 |
| 辣度 | 不辣 / 微辣 / 能吃辣 / 不限 |

筛选条件只对当前抽取流程生效，退出后恢复默认值。

## 6. 翻牌与再次抽取

### 6.1 翻牌动画

- 使用多张品类卡片快速切换的形式。
- 最终停留在一个品类结果。
- 动画时长控制在 1-2 秒，避免拖慢决策。

### 6.2 再随机一次

- 结果页提供“再随机一次”按钮。
- 本轮已经揭晓过的品类暂时排除，避免连续重复。
- 当所有可用品类均已出现时，提示：

```text
换一轮口味
```

- 提示后自动清空本轮排除列表，重新开始抽取。
- 离开当前抽取流程后，不保留排除记录。

## 7. 结果页

### 7.1 品类结果

主文案示例：

```text
今天吃火锅
```

品类结果旁提供收藏按钮。

### 7.2 店铺列表

- 展示与当前结果匹配、仍在营业的附近店铺。
- 默认按距离从近到远排序。
- 最多展示 3 家。
- 当前距离范围内的匹配店铺不足 3 家时，优先扩大距离范围。
- 扩大距离后仍不足 3 家时，展示实际数量，不强行补足。

店铺卡片展示：

- 店名。
- 距离。
- 人均价格。
- 评分；自定义店铺无评分时显示“暂无评分”。
- 营业状态。
- 数据标签：模拟店铺显示“模拟数据”，用户添加店铺显示“自定义”。
- 收藏按钮。
- 导航按钮。

### 7.3 导航

- 点击导航按钮后直接调用系统地图。
- 首版不在 App 内实现地图页。
- 模拟店铺使用模拟坐标发起导航。

## 8. 收藏与自定义内容

### 8.1 收藏页

首页右上角使用收藏图标进入收藏页。

收藏页包含两个标签：

```text
收藏品类
收藏店铺
```

- 用户可以取消收藏。
- 收藏店铺卡片保留导航按钮。
- 自定义店铺与收藏店铺在同一列表中展示，并标记“自定义”。
- 收藏仅保存在本机。
- 首版收藏品类不参与推荐加权。

### 8.2 自定义入口

收藏页右上角提供菜单：

```text
添加店铺
管理自定义品类
```

### 8.3 自定义店铺

用户可添加普通店铺、路边摊或非中式餐饮店铺。

必填项：

- 名称。
- 一个或多个品类。
- 位置：首版仅支持“使用当前位置”。
- 人均价格。

选填项：

- 营业时间。

规则：

- 未填写营业时间时，默认视为营业中。
- 自定义店铺可以编辑和删除。
- 自定义店铺参与抽取、附近店铺匹配和距离排序。

### 8.4 自定义品类

系统内置品类以中式餐饮为主。用户可增加日料、披萨等新类型。

自定义品类字段：

- 品类名称。
- 适用餐段：早餐 / 午餐 / 晚餐，可多选。
- 食物感受：热乎 / 清爽，可多选。
- 辣度：不辣 / 微辣 / 能吃辣，可多选。
- 早餐特征：方便带走 / 更管饱，可选。

规则：

- 自定义品类可编辑和删除。
- 添加店铺时既可选择已有品类，也可当场创建新品类。
- 删除仍被店铺使用的品类前，需要提示用户。
- 删除品类只解除店铺与该品类的关联，不删除店铺。
- 店铺失去全部品类关联时，提示用户重新选择品类。

## 9. 内置餐饮品类

首版主要覆盖常见中式餐饮。建议使用以下基础品类：

### 9.1 早餐

- 包子粥铺。
- 粉面。
- 饺子馄饨。
- 豆浆油条。

### 9.2 午餐与晚餐

- 火锅。
- 烧烤。
- 川湘菜。
- 家常炒菜。
- 粤菜。
- 麻辣烫。
- 粉面。
- 饺子馄饨。
- 砂锅煲仔饭。
- 快餐简餐。

同一家店可以关联多个品类。

## 10. 数据与地图适配

### 10.1 首版模拟数据

- 内置 20-30 家模拟店铺。
- 模拟店铺围绕稳定的模拟中心点生成约 100 米至 3 公里范围内的数据。
- 店铺名称、坐标、品类、距离、人均价格、评分和营业时间均通过统一数据模型提供。
- 模拟店铺明确标记“模拟数据”，避免误导用户。

### 10.2 后续真实地图接入

推荐逻辑不直接依赖具体地图服务。地图数据需要先转换成 App 内统一店铺模型，再进入推荐流程。

```text
模拟地图数据 ─┐
              ├→ 店铺数据适配层 → 统一店铺模型 → 推荐逻辑
真实地图数据 ─┘
```

真实地图服务可能无法提供完整菜单。后续接入时，允许通过地图分类、关键词映射和 App 内置品类标签补充店铺品类信息。

### 10.3 统一店铺模型

```text
Restaurant
- id
- sourceType: mock / custom / map
- name
- latitude
- longitude
- categoryIds[]
- averagePrice
- rating?
- businessHours?
- isOpen
```

### 10.4 统一品类模型

```text
FoodCategory
- id
- sourceType: builtIn / custom
- name
- mealPeriods[]
- foodFeelings[]
- spiceLevels[]
- breakfastTraits[]
```

## 11. 推荐与异常处理

### 11.1 基本推荐规则

- 只使用附近、仍在营业中的店铺。
- 附近不存在匹配店铺的品类不能被抽中。
- 一个品类即使只有 1 家可用店铺，也允许被抽中。
- 两种模式均可应用用户主动设置的折叠筛选条件。
- 回答两题模式额外应用问题答案。
- 首版不使用收藏、评分或天气进行复杂加权。

### 11.2 条件过严

筛选后没有可用结果时，自动逐步放宽：

```text
距离 → 预算 → 辣度
```

同时显示简短提示：

```text
附近选择不多，已经为你扩大范围
```

### 11.3 1.0 定位范围调整

- 1.0 使用稳定模拟位置完成核心体验，不申请系统定位权限。
- 真实定位、定位权限申请、拒绝授权强制退出和系统设置引导统一后置到 2.0。
- 代码中可以保留定位能力预研，但默认 1.0 产物不得要求用户授权。

## 12. 视觉方向

- 风格：活泼明亮，但不过度复杂。
- 色彩：以暖色为主，突出聚餐与食物氛围。
- 页面：保持少量核心元素，避免复杂层级。
- 动效：只为翻牌揭晓服务，不增加无意义动画。

## 13. MVP 页面清单

| 页面或状态 | 说明 |
| --- | --- |
| 首页 | 左右滑动切换两种抽取模式，包含折叠筛选入口和收藏入口 |
| 首页问答状态 | 在首页内完成两题，不额外跳转页面 |
| 翻牌状态 | 1-2 秒品类卡片快速切换动画 |
| 结果页 | 显示“今天吃某品类”、附近店铺、收藏、导航和再次抽取 |
| 收藏页 | 查看和取消收藏，进入自定义内容管理 |
| 添加或编辑店铺页 | 管理用户自定义店铺 |
| 自定义品类管理页 | 添加、编辑和删除自定义品类 |

## 14. MVP 验收标准

### 14.1 模拟位置

- 1.0 无需定位授权即可进入抽取流程。
- 模拟店铺围绕稳定模拟中心点生成，距离覆盖约 100 米至 3 公里。
- 真实定位验收后置到 2.0。

### 14.2 直接抽取

- 用户可在首页直接开始抽取。
- 系统按当前餐段和可用店铺过滤品类。
- 翻牌动画在 1-2 秒内完成。
- 结果页能够显示一个品类和最多 3 家匹配店铺。

### 14.3 回答两题

- 用户可左右滑动或点击标签切换到问答模式。
- 系统能够根据餐段展示固定的两个问题。
- 问题答案会影响候选品类。
- 完成两题后可正常进入翻牌与结果页。

### 14.4 再次抽取

- 再次抽取不会立即出现本轮已经揭晓过的品类。
- 候选品类耗尽后提示“换一轮口味”，并自动重置候选池。

### 14.5 筛选与放宽

- 距离、预算和辣度筛选默认收起。
- 用户选择的筛选条件能够影响抽取结果。
- 无匹配结果时按距离、预算、辣度顺序自动放宽，并显示提示。

### 14.6 收藏

- 用户可以收藏和取消收藏品类。
- 用户可以收藏和取消收藏店铺。
- 系统地图导航入口可保留，但属于试验性能力，不阻塞 1.0 结档。

### 14.7 自定义内容

- 用户能够添加、编辑和删除自定义店铺。
- 用户能够添加、编辑和删除自定义品类。
- 自定义店铺能够参与抽取、展示和距离排序。
- 删除被使用中的品类时有明确提示，不会误删关联店铺。

### 14.8 数据替换能力

- 推荐逻辑只依赖统一店铺模型。
- 模拟数据与未来地图数据通过适配层接入。
- 更换数据来源时，不需要重写核心抽取流程。

## 2. 视觉概念

### 2.1 关键词

```text
暖色
轻松
有食欲
有一点游戏感
信息克制
```

### 2.2 设计原则

- 首页第一眼只需要回答一件事：用户可以快速开始决定“今天吃什么”。
- 主要行动按钮必须醒目，但页面不使用过多高饱和色块。
- 页面使用暖白背景、橙红主色和少量浅黄色点缀，避免做成外卖平台的信息密集风格。
- 通过圆角卡片和轻量插画占位营造轻松感，首版不依赖复杂图片素材。
- 不使用底部导航栏。

# 今天吃什么 App - M3 UI 回答两题与翻牌体验

## 1. 当前交付

本文档是 UI Agent 在 M3“回答两题与游戏化体验”阶段的交付物。

M2 已通过产品验收。M3 在现有首页、筛选区域和结果页基础上补充：


本阶段不增加独立问答页、不增加转盘、不增加无意义动效，也不扩展到收藏、自定义内容或定位权限流程。

## 2. 体验原则

- 问答模式只比直接抽取多出 2 次轻量选择，不增加复杂配置。
- 两道问题都在首页主卡片内完成，不跳转新页面。
- 用户始终知道自己正在回答第几题，并且可以返回修改上一题。
- 翻牌动画只服务于结果揭晓，总时长控制在 1-2 秒。
- 直接抽取和回答两题共用同一翻牌状态，避免出现两套视觉语言。
- 折叠筛选继续保留在主卡片下方，默认收起。

# 鸿蒙系统App AI开发流程（Codex可执行版）

## 1. 系统开发总体流程
1. **里程碑规划**
   - 定义产品迭代节点（Milestones）
   - 每个节点包含：
     - 目标功能
     - 验收标准（功能可运行/性能基本稳定/初步UI）
2. **最小可行产品 (MVP)**
   - 快速验证App能否运行
   - 功能最简化，先保证“能跑”
   - 外观和体验暂时次要
3. **Spec文档输出**
   - 根据需求文档自动生成详细Spec
   - Spec内容包括：
     - 功能模块描述
     - 接口说明
     - 数据结构
     - UI布局参考
     - 运行环境和兼容性要求
4. **AI角色开发执行**
   - 按照Spec文档开发
   - AI根据角色职责完成任务
   - 定期验收并更新里程碑

---

## 2. 产品开发角色定义
| 角色 | 职责 | 输出文档/结果 |
|------|------|---------------|
| **产品Agent** | 规划里程碑、生成Spec、验收标准 | 里程碑文档、Spec文档、验收报告 |
| **研发Agent** | 按Spec开发App | 功能模块、可运行App原型 |
| **UIAgent** | 提供视觉参考、界面风格 | UI素材、布局方案、风格指导 |

---

## 3. 里程碑示例（Milestones）
1. **Milestone 1: 基础MVP**
   - 功能：App能启动、基础界面可显示
   - 验收标准：空白界面可启动，按钮可点击
2. **Milestone 2: 核心功能实现**
   - 功能：根据用户需求显示随机餐厅推荐
   - 验收标准：随机功能可用，基础逻辑可运行
3. **Milestone 3: 智能化选择**
   - 功能：根据用户喜好、天气、评分推荐
   - 验收标准：推荐结果合理，可反馈评分
4. **Milestone 4: UI优化**
   - 功能：界面美观、交互顺畅
   - 验收标准：用户可操作无障碍，风格统一

---

## 4. Spec文档结构
- **功能模块**
  - 模块名
  - 描述
  - 输入/输出
  - 依赖关系
- **界面布局**
  - 页面名称
  - 元素列表
  - 交互行为
- **数据结构**
  - 数据字段
  - 类型
  - 描述
- **运行环境**
  - 鸿蒙系统版本
  - 支持屏幕尺寸
  - 权限需求
- **接口规范**
  - API接口名
  - 参数说明
  - 返回值说明

---

## 5. Codex执行策略
1. **初始化**
   - 产品Agent输出里程碑与Spec
2. **按里程碑迭代**
   - 每个Milestone由研发Agent和UIAgent执行
   - 研发Agent按Spec开发功能
   - UIAgent提供界面和风格素材
3. **验收**
   - 产品Agent检查功能是否满足Spec
   - 若不满足，生成Bug报告并返回研发Agent修改
4. **迭代优化**
   - 迭代更新Spec和UI
   - 完成最终产品
